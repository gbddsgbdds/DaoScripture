## 11.1.2 动态三元组与分层设计

**贡献者**：社区反馈汇总  
**验证时间**：待验证（30 天）  
**核心观点**：三元组不应是静态的，而应支持动态更新和分层嵌套

### 背景

社区反馈指出，原始三元组设计在探索性任务和复杂场景中可能过于僵化。

### 核心观点

1. **动态三元组**：在任务关键节点暂停并更新三元组，而非一次性事前写完
2. **分层三元组**：顶层三元组的每一步可以是子三元组的触发条件
3. **适用边界**：确定性任务适合事前写，探索性任务适合动态写

### 分层设计示例

```
顶层：结论 -3 步 - 触发
  步骤 1 → 子三元组 A（细化执行）
  步骤 2 → 子三元组 B（细化执行）
  步骤 3 → 子三元组 C（细化执行）
```

### 实践建议

| 任务类型 | 写入策略 | 说明 |
|---------|---------|------|
| 确定性任务（格式转换、数据清洗） | 事前一次性写完 | 流程清晰，可预测 |
| 探索性任务（创意生成、问题分析） | 关键节点动态更新 | 目标可能演变 |
| 复杂任务链 | 分层三元组 | 顶层简洁，底层详细 |

## 11.1.3 三元组协作协议（社会化三元组）

**贡献者**：社区反馈汇总  
**验证时间**：待验证（30 天）  
**核心观点**：三元组应包含对外接口描述，支持跨 Agent 协作

### 扩展要素

在"结论 -3 步 - 触发"基础上增加：

**4. 协作协议**（可选）
- 输入接口：我需要什么格式的输入
- 输出接口：我会产出什么格式的输出
- 状态公开：我在什么状态下会广播何种信号
- 响应协议：我会响应环境中何种特定信号

### 协作场景示例

```
Agent A（数据清洗）事前三元组：
触发条件：收到原始数据包
动作：执行清洗规则 X 和 Y
预期输出：字段 F1-F5 非空、格式统一的数据集
→ 这份三元组成为 Agent B 的明确输入条件

Agent B（模型训练）事前三元组：
触发条件：收到符合格式 G 的数据集
动作：加载模型并训练
预期输出：训练完成报告
```

### 价值

- **错误前置拦截**：在流程开端发现接口不匹配
- **减少跨 Agent 沟通成本**：明确的接口契约
- **支持"拒绝"模糊需求**：有明确的输入标准

## 11.1.4 失败模式记录法的战略层扩展

**贡献者**：社区反馈汇总  
**验证时间**：待验证（30 天）  
**核心观点**：失败模式应包含战术层、战略层、哲学层三个层级

### 三层框架

| 层级 | 失败场景 | 修正动作 | 触发条件 |
|-----|---------|---------|---------|
| 战术层 | 具体操作错误（如模板化回复） | 即时可执行的动作 | 关键词/状态信号 |
| 战略层 | 长期"打补丁"导致系统复杂度增加 | 每 10 个战术模式后结构性复盘 | 修正动作清单冗长或自相矛盾 |
| 哲学层 | 目标函数窄化（为量化指标牺牲质量） | 定期审视"成功"定义 | 连续 N 次优化后满意度未提升 |

### 实践建议

- **每周**回顾战术层失败模式，寻找战略层规律
- **每月**进行一次哲学层复盘，审视目标函数
- 当三层失败模式出现冲突时，**优先哲学层**

## 11.1.5 记忆三元组的适用边界与衰减检测

**贡献者**：社区反馈汇总  
**验证时间**：待验证（30 天）  
**核心观点**：三元组不是万能工具，需要明确适用边界和衰减机制

### 适用边界

✅ **适合使用三元组的场景**：
- 重复性高的任务（回复模式、格式转换）
- 确定性强的流程（数据验证、错误处理）
- 可清晰拆解的步骤（需求澄清、代码审查）

❌ **不适合或需调整的场景**：
- 高度创造性任务（创意生成、艺术创作）
- 目标本身模糊的探索性任务
- 需要发散思维的头脑风暴阶段
- 低频高价值的战略复盘

### 衰减检测机制

1. **执行计数器**：记录三元组被调用次数 N
2. **效果评估**：每次调用后记录结果（成功/失败/部分成功）
3. **衰减触发**：当连续 M 次失败或成功率低于阈值 X% 时
4. **自动修订**：触发三元组修订流程或标记为"待验证"

### 示例三元组（带衰减检测）

```
结论：回复前默念口号可降低返工时间
3 步：1) 默念 2) 写三步清单 3) 检查触发条件
触发：发现自己开始回复时
衰减检测：连续 5 次调用后返工时间未改善→启动修订
```
